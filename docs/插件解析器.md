# 插件解析器设计

## 1. 概述

`PluginParser` 从上传的 ZIP/JAR 文件中定位并解析 `plugin.xml`，提取所有元数据。实现"零配置发布"：开发者只需上传文件，不需要手动填写任何元数据。

**关键约束：**

- 输入是 `File` 对象（来自 multipart/form-data）
- 必须支持两种 JetBrains 插件包格式（JAR 和 ZIP）
- 解析过程是同步的（adm-zip 内存操作）
- 所有元数据从 `plugin.xml` 中提取，不接受外部参数

## 2. JetBrains 插件包格式

### 2.1 格式 A — 单 JAR 文件

无第三方依赖的简单插件，文件本身就是 JAR（JAR 本质是 ZIP 格式）：

```
my-plugin.jar
├── META-INF/
│   ├── plugin.xml          ← 目标
│   ├── pluginIcon.svg      ← 可选：插件图标
│   └── MANIFEST.MF
└── com/example/plugin/
    └── MyAction.class
```

**识别方式：** `zip.getEntry("META-INF/plugin.xml")` 直接找到。

### 2.2 格式 B — ZIP 包（含 lib 目录）

有依赖的插件，由 `buildPlugin` Gradle task 生成：

```
my-plugin-1.0.0.zip
└── my-plugin/                        ← 顶层目录名 = 插件名
    └── lib/
        ├── my-plugin-1.0.0.jar       ← plugin.xml 在这个 JAR 里
        ├── kotlin-stdlib-1.9.0.jar   ← 依赖 JAR（无 plugin.xml）
        ├── kotlinx-coroutines-1.7.jar
        └── other-lib.jar
```

**识别方式：** 顶层无 `META-INF/plugin.xml`，需要遍历 `*/lib/*.jar`，打开每个 JAR 寻找 `META-INF/plugin.xml`。

### 2.3 格式差异对比

| | 格式 A (JAR) | 格式 B (ZIP) |
|--|-------------|-------------|
| 文件扩展名 | `.jar` | `.zip` |
| plugin.xml 位置 | 根 `META-INF/plugin.xml` | `*/lib/*.jar` 内 |
| 产生方式 | 简单插件直接打包 | `buildPlugin` Gradle task |
| 是否有依赖 | 无外部依赖 | 有第三方依赖 |
| 解析复杂度 | 简单 | 需要嵌套 ZIP 读取 |

## 3. 解析流程

```
输入: 临时文件路径 (tmpPath)
          │
          ▼
  [1] 文件大小校验
      > config.maxFileSize → 抛 InvalidPluginError
          │
          ▼
  [2] 用 adm-zip 打开外层 ZIP
      打开失败 → 抛 InvalidPluginError("Not a valid ZIP/JAR file")
          │
          ▼
  [3] 查找 META-INF/plugin.xml 条目
          │
    ┌─────┴─────┐
    ▼           ▼
  找到        没找到
  (JAR格式)   (ZIP格式)
    │           │
    │           ▼
    │     [4] 查找 */lib/*.jar 条目
    │         按路径过滤：
    │         - 不是目录
    │         - 路径深度 ≥ 3 段
    │         - 倒数第二段 = "lib"
    │         - 以 .jar 结尾
    │           │
    │           ▼
    │     [5] 对每个 jar 条目：
    │         getData() → adm-zip 打开
    │         查找其中的 META-INF/plugin.xml
    │         找到第一个即停止
    │         打开失败 → 跳过该 jar
    │           │
    │         全部都没找到 → 抛 InvalidPluginError
    │           │
    └─────┬─────┘
          ▼
  [6] 读取 plugin.xml 文本（UTF-8）
          │
          ▼
  [7] 正则解析 XML 提取字段
      - extractTag: <tag>text</tag>
      - extractCdata: <tag><![CDATA[...]]></tag>
      - extractAttr: <tag attr="value" />
      - extractVendor: <vendor url="..." email="...">name</vendor>
      - extractAllTags: 多个同名标签
          │
          ▼
  [8] 校验必填字段
      缺少 id → InvalidPluginError("Missing required element: <id>")
      缺少 name → InvalidPluginError("Missing required element: <name>")
      缺少 version → InvalidPluginError("Missing required element: <version>")
      缺少 since-build → InvalidPluginError("Missing required attribute: idea-version since-build")
          │
          ▼
  [9] 安全校验
      id 包含 ".." "/" "\" → InvalidPluginError("Invalid id: must not contain path separators")
      version 包含 ".." "/" "\" → 同上
          │
          ▼
  [10] 返回 PluginMetadata
```

## 4. plugin.xml 完整示例

### 4.1 最简 plugin.xml（必填字段）

```xml
<idea-plugin>
  <id>com.example.myplugin</id>
  <name>My Plugin</name>
  <version>1.0.0</version>
  <idea-version since-build="222"/>
</idea-plugin>
```

### 4.2 完整 plugin.xml（所有字段）

```xml
<idea-plugin>
  <id>com.example.myplugin</id>
  <name>My Custom Plugin</name>
  <version>1.2.0</version>

  <vendor email="dev@example.com" url="https://example.com">Example Inc</vendor>

  <description><![CDATA[
    <h2>My Custom Plugin</h2>
    <p>A powerful plugin for IntelliJ IDEA that adds:</p>
    <ul>
      <li>Feature A</li>
      <li>Feature B</li>
    </ul>
  ]]></description>

  <change-notes><![CDATA[
    <h3>v1.2.0</h3>
    <ul>
      <li>Added Feature B</li>
      <li>Fixed crash on startup</li>
    </ul>
  ]]></change-notes>

  <idea-version since-build="222" until-build="241.*"/>

  <depends>com.intellij.modules.platform</depends>
  <depends>com.intellij.modules.lang</depends>
  <depends optional="true" config-file="yaml-support.xml">org.jetbrains.plugins.yaml</depends>
</idea-plugin>
```

### 4.3 IntelliJ Platform Plugin 2.x 的 plugin.xml

IntelliJ Platform Gradle Plugin 2.x 生成的插件可能使用不同的命名空间或属性，但核心元素结构不变：

```xml
<idea-plugin xmlns:xi="http://www.w3.org/2001/XInclude">
  <id>com.example.myplugin</id>
  <name>My Plugin</name>
  <version>2.0.0</version>
  <idea-version since-build="231" until-build="242.*"/>
  <depends>com.intellij.modules.platform</depends>
</idea-plugin>
```

解析器忽略命名空间声明和未知属性，只关注目标元素和属性。

## 5. plugin.xml 字段映射

| plugin.xml 路径 | PluginMetadata 字段 | 必填 | 值类型 | 说明 |
|----------------|---------------------|------|--------|------|
| `/idea-plugin/id` | `id` | 是 | 纯文本 | 唯一标识，如 `com.example.myplugin` |
| `/idea-plugin/name` | `name` | 是 | 纯文本 | 显示名称 |
| `/idea-plugin/version` | `version` | 是 | 纯文本 | 语义化版本号 |
| `/idea-plugin/idea-version/@since-build` | `sinceBuild` | 是 | 属性值 | 最低兼容 IDE build |
| `/idea-plugin/idea-version/@until-build` | `untilBuild` | 否 | 属性值 | 最高兼容 IDE build |
| `/idea-plugin/description` | `description` | 否 | 纯文本或 CDATA | 可含 HTML |
| `/idea-plugin/vendor` (text) | `vendor.name` | 否 | 纯文本 | 厂商名称 |
| `/idea-plugin/vendor/@url` | `vendor.url` | 否 | 属性值 | 厂商网址 |
| `/idea-plugin/vendor/@email` | `vendor.email` | 否 | 属性值 | 厂商邮箱 |
| `/idea-plugin/change-notes` | `changeNotes` | 否 | 纯文本或 CDATA | 可含 HTML |
| `/idea-plugin/depends` (所有) | `depends[]` | 否 | 纯文本数组 | 插件依赖列表 |

### 5.1 CDATA 与纯文本的区别

`description` 和 `change-notes` 可能是两种格式：

```xml
<!-- 格式 1：纯文本 -->
<description>A simple plugin.</description>

<!-- 格式 2：CDATA（包含 HTML） -->
<description><![CDATA[<p>A <b>rich</b> plugin.</p>]]></description>
```

解析器同时尝试两种格式，优先匹配 CDATA：

```typescript
const description = this.extractCdata(xmlContent, "description")
  ?? this.extractTag(xmlContent, "description");
```

### 5.2 depends 标签的特殊处理

`<depends>` 可能有 `optional` 属性：

```xml
<depends>com.intellij.modules.platform</depends>
<depends optional="true" config-file="yaml.xml">org.jetbrains.plugins.yaml</depends>
```

解析器提取所有 `<depends>` 标签的文本内容，不区分 optional：

```typescript
const depends = this.extractAllTags(xmlContent, "depends");
// → ["com.intellij.modules.platform", "org.jetbrains.plugins.yaml"]
```

## 6. 实现

```typescript
// src/service/plugin-parser.ts
import AdmZip from "adm-zip";
import type { PluginMetadata, VendorInfo } from "../repository/types";

export class InvalidPluginError extends Error {
  constructor(message: string) {
    super(message);
    this.name = "InvalidPluginError";
  }
}

export class PluginParser {

  /**
   * 从 ZIP/JAR 文件中解析 plugin.xml，提取元数据。
   *
   * @param filePath 临时文件路径
   * @returns 解析出的插件元数据
   * @throws InvalidPluginError 文件格式无效、找不到 plugin.xml、缺少必填字段
   */
  parse(filePath: string): PluginMetadata {
    let zip: AdmZip;
    try {
      zip = new AdmZip(filePath);
    } catch (e) {
      throw new InvalidPluginError(
        `Not a valid ZIP/JAR file: ${e instanceof Error ? e.message : "unknown error"}`
      );
    }

    // 尝试方式 1: 作为 JAR 直接查找 plugin.xml
    const directEntry = zip.getEntry("META-INF/plugin.xml");
    if (directEntry) {
      const xmlContent = directEntry.getData().toString("utf-8");
      return this.parsePluginXml(xmlContent);
    }

    // 尝试方式 2: 在嵌套的 lib/*.jar 中查找
    const jarEntries = zip.getEntries().filter((entry) => {
      const parts = entry.entryName.split("/");
      return (
        !entry.isDirectory &&
        parts.length >= 3 &&
        parts[parts.length - 2] === "lib" &&
        entry.entryName.endsWith(".jar")
      );
    });

    if (jarEntries.length === 0) {
      throw new InvalidPluginError(
        "No plugin.xml found. " +
        "Expected either META-INF/plugin.xml in root (JAR format), " +
        "or */lib/*.jar containing META-INF/plugin.xml (ZIP format)."
      );
    }

    for (const jarEntry of jarEntries) {
      const jarData = jarEntry.getData();
      try {
        const innerZip = new AdmZip(jarData);
        const pluginXmlEntry = innerZip.getEntry("META-INF/plugin.xml");
        if (pluginXmlEntry) {
          const xmlContent = pluginXmlEntry.getData().toString("utf-8");
          return this.parsePluginXml(xmlContent);
        }
      } catch {
        // 该 jar 无法作为 zip 打开（可能是损坏的 JAR），跳过
        continue;
      }
    }

    throw new InvalidPluginError(
      `Searched ${jarEntries.length} JAR file(s) in lib/ directory, ` +
      "but none contained META-INF/plugin.xml.\n" +
      "JARs searched: " + jarEntries.map(e => e.entryName).join(", ")
    );
  }

  /**
   * 解析 plugin.xml 内容，提取所有字段。
   */
  private parsePluginXml(xmlContent: string): PluginMetadata {
    // 必填字段
    const id = this.extractTag(xmlContent, "id");
    if (!id) {
      throw new InvalidPluginError(
        "Missing required element: <id>\n" +
        "plugin.xml must contain: <id>com.example.myplugin</id>"
      );
    }

    const name = this.extractTag(xmlContent, "name");
    if (!name) {
      throw new InvalidPluginError(
        "Missing required element: <name>\n" +
        "plugin.xml must contain: <name>My Plugin Name</name>"
      );
    }

    const version = this.extractTag(xmlContent, "version");
    if (!version) {
      throw new InvalidPluginError(
        "Missing required element: <version>\n" +
        "plugin.xml must contain: <version>1.0.0</version>"
      );
    }

    const sinceBuild = this.extractAttr(xmlContent, "idea-version", "since-build");
    if (!sinceBuild) {
      throw new InvalidPluginError(
        "Missing required attribute: idea-version since-build\n" +
        "plugin.xml must contain: <idea-version since-build=\"222\"/>"
      );
    }

    // 可选字段
    const untilBuild = this.extractAttr(xmlContent, "idea-version", "until-build");
    const description = this.extractCdata(xmlContent, "description")
      ?? this.extractTag(xmlContent, "description");
    const changeNotes = this.extractCdata(xmlContent, "change-notes")
      ?? this.extractTag(xmlContent, "change-notes");

    const vendor = this.extractVendor(xmlContent);
    const depends = this.extractAllTags(xmlContent, "depends");

    // 安全校验
    this.validateSafePath(id, "id");
    this.validateSafePath(version, "version");

    return {
      id, name, version, sinceBuild, untilBuild,
      description, vendor, changeNotes, depends,
    };
  }

  // ============ XML 提取方法 ============

  /**
   * 提取简单标签文本：<tag>content</tag>
   *
   * 匹配规则：
   * - 标签名精确匹配
   * - 允许标签有属性（如 <depends optional="true">）
   * - 提取闭合标签之间的纯文本（不含子标签）
   */
  private extractTag(xml: string, tag: string): string | null {
    const match = xml.match(new RegExp(`<${tag}[^>]*>([^<]+)</${tag}>`));
    return match?.[1]?.trim() ?? null;
  }

  /**
   * 提取 CDATA 内容：<tag><![CDATA[content]]></tag>
   *
   * 用于 description 和 change-notes，这两个字段通常包含 HTML。
   * CDATA 中可以包含任意字符（包括 < > &），不受 XML 转义限制。
   */
  private extractCdata(xml: string, tag: string): string | null {
    const match = xml.match(
      new RegExp(`<${tag}[^>]*>\\s*<!\\[CDATA\\[([\\s\\S]*?)\\]\\]>\\s*</${tag}>`)
    );
    return match?.[1]?.trim() ?? null;
  }

  /**
   * 提取自闭合标签或空标签的属性值：<tag attr="value" />
   *
   * 用于提取 idea-version 的 since-build 和 until-build 属性。
   *
   * 匹配两种格式：
   *   <idea-version since-build="222" until-build="241.*"/>
   *   <idea-version since-build="222" until-build="241.*" />
   */
  private extractAttr(xml: string, tag: string, attr: string): string | null {
    const tagMatch = xml.match(new RegExp(`<${tag}\\s+([^>]*)/?>`));
    if (!tagMatch) return null;
    const attrMatch = tagMatch[1].match(new RegExp(`${attr}="([^"]*)"`));
    return attrMatch?.[1]?.trim() || null;
  }

  /**
   * 提取所有同名标签的文本
   *
   * 用于 <depends> 标签，一个插件可能有多个依赖。
   * 忽略带 optional 等属性的差异，只提取文本内容。
   */
  private extractAllTags(xml: string, tag: string): string[] {
    const matches = xml.matchAll(new RegExp(`<${tag}[^>]*>([^<]+)</${tag}>`, "g"));
    return Array.from(matches, (m) => m[1].trim());
  }

  /**
   * 提取 vendor 信息
   *
   * vendor 标签同时有属性和文本内容：
   *   <vendor email="dev@example.com" url="https://example.com">Example Inc</vendor>
   *
   * 其中 name 是标签文本，email 和 url 是属性。
   * 三者都是可选的（可能只有名称没有 email/url）。
   */
  private extractVendor(xml: string): VendorInfo | null {
    // 尝试匹配有属性的 vendor
    const withAttrs = xml.match(/<vendor\s+([^>]*)>([^<]*)<\/vendor>/);
    if (withAttrs) {
      const attrs = withAttrs[1];
      const name = withAttrs[2].trim();
      const url = attrs.match(/url="([^"]*)"/)?.[1] || null;
      const email = attrs.match(/email="([^"]*)"/)?.[1] || null;
      return { name, url, email };
    }

    // 尝试匹配无属性的 vendor：<vendor>Example Inc</vendor>
    const withoutAttrs = xml.match(/<vendor>([^<]+)<\/vendor>/);
    if (withoutAttrs) {
      return { name: withoutAttrs[1].trim(), url: null, email: null };
    }

    return null;
  }

  // ============ 安全校验 ============

  /**
   * 路径遍历防护
   *
   * pluginId 和 version 会被用于构建文件系统路径：
   *   data/plugins/{pluginId}/{version}.zip
   *
   * 必须防止路径遍历攻击：
   *   id = "../../etc/passwd"  → 拒绝
   *   id = "com.example/test"  → 拒绝
   *   id = "com.example\\test" → 拒绝
   */
  private validateSafePath(value: string, fieldName: string): void {
    if (value.includes("..") || value.includes("/") || value.includes("\\")) {
      throw new InvalidPluginError(
        `Invalid ${fieldName}: "${value}" must not contain path separators (.. / \\)`
      );
    }

    // 额外校验：不允许空值或纯空白
    if (value.trim().length === 0) {
      throw new InvalidPluginError(
        `Invalid ${fieldName}: must not be empty or whitespace-only`
      );
    }
  }
}
```

## 7. 为什么用正则而非 XML 解析器

| 方案 | 优点 | 缺点 |
|------|------|------|
| 正则提取 | 零依赖、轻量、够用 | 处理复杂嵌套有局限 |
| `fast-xml-parser` | 完整解析 | 额外依赖 (~50KB) |
| DOM/SAX 解析 | 标准、健壮 | Bun 无内置 DOM |

选择正则的理由：

1. `plugin.xml` 结构固定，JetBrains 定义了严格的 schema
2. 只需提取 7 个元素/属性，不需要遍历 DOM
3. 不执行动态代码，天然防 XML 注入
4. 零额外依赖

**已知限制：** 如果 `<name>` 或 `<id>` 标签内包含子标签（如 `<name><b>My</b> Plugin</name>`），正则匹配会失败。但这种写法在 plugin.xml 中不合法，JetBrains 自己也不支持。

## 8. 安全考虑

### 8.1 ZIP 炸弹防护

adm-zip 在内存中操作整个 ZIP 内容。恶意用户可能上传压缩比极高的 ZIP 文件（ZIP Bomb）。

**防御层次：**

```
层次 1: Hono 层（请求体大小限制）
  Bun.serve 默认限制请求体大小，但 multipart 可能绕过

层次 2: PluginService.upload()（文件大小校验）
  ┌──────────────────────────────────────────┐
  │ const file = body.get("file") as File;   │
  │ if (file.size > config.maxFileSize) {     │
  │   throw new InvalidPluginError(           │
  │     `File too large: ${file.size} bytes`  │
  │     ` (max: ${config.maxFileSize})`       │
  │   );                                      │
  │ }                                         │
  └──────────────────────────────────────────┘

层次 3: 临时文件写入后校验
  ┌──────────────────────────────────────────┐
  │ await Bun.write(tmpPath, file);          │
  │ const stat = await Bun.file(tmpPath)     │
  │   .stat();                               │
  │ if (stat.size > config.maxFileSize) {    │
  │   await unlink(tmpPath);                 │
  │   throw new InvalidPluginError(          │
  │     "File too large"                     │
  │   );                                      │
  │ }                                         │
  └──────────────────────────────────────────┘
```

默认限制 `MAX_FILE_SIZE=100MB`（可通过环境变量调整）。

### 8.2 路径遍历防护

```
攻击场景：
  plugin.xml 中  <id>../../etc</id>  <version>passwd</version>
  构建路径：data/plugins/../../etc/passwd.zip
  实际路径：data/etc/passwd.zip  ← 路径遍历！

防御：
  validateSafePath() 拒绝包含 ".." "/" "\" 的值
```

### 8.3 XML 注入防护

**解析层面：** 使用正则提取而非 eval/DOM，不执行任何动态代码。即使 plugin.xml 包含恶意内容（如 XXE 实体注入），正则解析也不会触发。

**存储/展示层面：** `description` 和 `changeNotes` 中的 HTML 内容：
- 原样存储到数据库（JetBrains IDE 需要原始 HTML）
- Web UI 渲染前做 HTML 净化（移除 `<script>` 等危险标签）
- API 返回时不做转义（JSON 本身会转义特殊字符）

### 8.4 嵌套 JAR 安全

打开嵌套 JAR 时可能遇到：
- 损坏的 JAR 文件 → `try/catch` 跳过
- 超大 JAR 解压后内存溢出 → 依赖前置的文件大小限制
- 恶意的 JAR 条目名（路径遍历）→ adm-zip 不解压到文件系统，只在内存操作

## 9. 错误信息设计

每个错误都提供足够上下文帮助用户定位问题：

| 场景 | 错误信息 |
|------|---------|
| 不是 ZIP/JAR | `Not a valid ZIP/JAR file: <原始错误>` |
| 根目录无 plugin.xml，也无 lib/*.jar | `No plugin.xml found. Expected either META-INF/plugin.xml in root (JAR format), or */lib/*.jar containing META-INF/plugin.xml (ZIP format).` |
| lib/ 中的 JAR 都没有 plugin.xml | `Searched N JAR file(s) in lib/ directory, but none contained META-INF/plugin.xml. JARs searched: a.jar, b.jar, c.jar` |
| 缺少 `<id>` | `Missing required element: <id>. plugin.xml must contain: <id>com.example.myplugin</id>` |
| 缺少 `<name>` | `Missing required element: <name>. plugin.xml must contain: <name>My Plugin Name</name>` |
| 缺少 `<version>` | `Missing required element: <version>. plugin.xml must contain: <version>1.0.0</version>` |
| 缺少 `since-build` | `Missing required attribute: idea-version since-build. plugin.xml must contain: <idea-version since-build="222"/>` |
| id 含路径分隔符 | `Invalid id: "../../etc" must not contain path separators (.. / \)` |
| version 含路径分隔符 | `Invalid version: "1.0/0" must not contain path separators (.. / \)` |

## 10. 与 PluginService 的集成

```typescript
// src/service/plugin-service.ts（upload 方法中调用解析器的部分）

/** 版本冲突错误，用于区分 409 Conflict 响应 */
export class ConflictError extends Error {
  constructor(message: string) {
    super(message);
    this.name = "ConflictError";
  }
}

async upload(file: File, force: boolean): Promise<PluginVersion> {
  // 1. 写入临时文件
  const tmpPath = join(this.storageDir, "tmp", crypto.randomUUID());
  await mkdir(dirname(tmpPath), { recursive: true });
  await Bun.write(tmpPath, file);

  try {
    // 2. 解析元数据
    const metadata = this.parser.parse(tmpPath);

    // 3. 检查版本冲突
    const existing = this.repository.findVersion(metadata.id, metadata.version);
    if (existing && !force) {
      throw new ConflictError(
        `Version ${metadata.version} already exists for ${metadata.id}. ` +
        `Use ?force=true to overwrite.`
      );
    }

    // 4. 移动到目标路径
    const targetDir = join(this.storageDir, "plugins", metadata.id);
    await mkdir(targetDir, { recursive: true });
    const targetPath = join(targetDir, `${metadata.version}.zip`);
    await Bun.write(targetPath, Bun.file(tmpPath));

    // 5. 计算文件信息
    const hasher = new Bun.CryptoHasher("sha256");
    hasher.update(await Bun.file(targetPath).arrayBuffer());
    const fileInfo: FileInfo = {
      fileName: `${metadata.id}-${metadata.version}.zip`,
      fileSize: file.size,
      fileSha256: hasher.digest("hex"),
    };

    // 6. 保存到数据库
    this.repository.savePluginVersion(metadata, fileInfo);

    return this.repository.findVersion(metadata.id, metadata.version)!;
  } finally {
    // 7. 清理临时文件
    try { await unlink(tmpPath); } catch { /* ignore */ }
  }
}
```

## 11. 性能考虑

| 场景 | 影响 | 处理 |
|------|------|------|
| 大 ZIP 文件（50MB+） | adm-zip 在内存中加载整个文件 | 前置 MAX_FILE_SIZE 限制（默认 100MB） |
| 多个 JAR 嵌套 | 每个 JAR 都需要 getData() + new AdmZip() | 找到第一个 plugin.xml 即停止 |
| 频繁上传 | 临时文件可能堆积 | finally 块确保清理；可加定时清理 |

典型插件包大小：1-20MB。解析时间 < 100ms。内存峰值约为 ZIP 文件大小的 2-3 倍（压缩数据 + 解压数据）。
