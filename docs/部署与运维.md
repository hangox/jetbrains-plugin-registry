# 部署与运维

## 1. 快速开始（开发者上手）

### 1.1 环境要求

- [Bun](https://bun.sh) >= 1.1（推荐 1.2+）
- SQLite（Bun 内置，无需单独安装）

### 1.2 初始化项目

```bash
# 克隆仓库
git clone <repo-url>
cd jetbrains-plugin-registry

# 安装服务端依赖
cd server
bun install

# 创建 .env 文件
cat > .env << 'EOF'
PORT=3000
BASE_URL=http://localhost:3000
AUTH_TOKENS=dev-token
ADMIN_USER=admin
ADMIN_PASS=admin123
DATA_DIR=./data
EOF

# 启动开发模式（热重载）
bun run dev
```

### 1.3 验证服务

```bash
# 健康检查
curl http://localhost:3000/api/health

# 访问 Web 管理界面
open http://localhost:3000

# 用 dev-token 测试上传（需要有一个 JetBrains 插件包）
curl -X POST http://localhost:3000/api/plugins \
  -H "Authorization: Bearer dev-token" \
  -F "file=@path/to/plugin.zip"
```

### 1.4 数据库初始化

首次启动时自动创建 SQLite 数据库和表结构。如需重置：

```bash
trash ./data/registry.db    # 删除数据库
bun run dev               # 重新启动自动建表
```

## 2. 构建与运行

### 2.1 开发模式

```bash
cd server
bun install
bun run dev    # 等同于 bun run --hot src/index.ts
```

### 2.2 生产模式

**方式 A：直接运行（推荐，最简单）**

```bash
cd server
bun install --production
AUTH_TOKENS=xxx ADMIN_PASS=your-password BASE_URL=https://plugins.example.com bun run src/index.ts
```

Bun 原生运行 TypeScript，不需要编译步骤。

**方式 B：编译为单文件可执行**

```bash
bun build src/index.ts --compile --outfile plugin-registry
# 产出单个二进制文件 plugin-registry (~50MB)
# 不需要 Bun 运行时，可以直接分发
./plugin-registry
```

### 2.3 内存控制

Bun 默认就很省内存（idle ~30MB）。如需进一步限制：

```bash
# 限制 V8 heap 上限为 64MB
bun --smol run src/index.ts
```

`--smol` 标志启用低内存模式，减少 GC 和 JIT 开销。

## 3. 环境变量

| 变量 | 必填 | 默认值 | 说明 |
|------|------|--------|------|
| `BASE_URL` | 是 | - | 公网地址，用于 XML 中的下载链接 |
| `AUTH_TOKENS` | 是 | - | 认证 Token，多个用逗号分隔 |
| `PORT` | 否 | `3000` | 监听端口 |
| `DATA_DIR` | 否 | `./data` | 数据存储目录 |
| `DB_TYPE` | 否 | `sqlite` | 数据库类型 |
| `DB_URL` | 否 | `registry.db` | SQLite 文件名或 JDBC URL |
| `ADMIN_USER` | 否 | `admin` | Web 管理界面用户名 |
| `ADMIN_PASS` | 是 | - | Web 管理界面密码 |
| `SESSION_SECRET` | 否 | 自动生成 | Cookie 签名密钥（重启后会失效，生产环境建议显式配置） |
| `MAX_FILE_SIZE` | 否 | `104857600` | 上传限制（字节，默认 100MB） |

Bun 原生支持 `.env` 文件，放在项目根目录即自动加载。

## 4. Docker 部署

### 4.1 Dockerfile

```dockerfile
FROM oven/bun:1-alpine

WORKDIR /app

# 安装依赖
COPY server/package.json server/bun.lock ./
RUN bun install --production --frozen-lockfile

# 复制源码
COPY server/src ./src
COPY server/tsconfig.json ./

# 数据目录
RUN mkdir -p /app/data
VOLUME /app/data

ENV DATA_DIR=/app/data
ENV PORT=3000

EXPOSE 3000

CMD ["bun", "run", "src/index.ts"]
```

镜像大小约 **80MB**（oven/bun:1-alpine ~70MB + 依赖 ~10MB）。

### 4.2 docker-compose.yml

```yaml
services:
  registry:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    volumes:
      - registry-data:/app/data
    environment:
      - BASE_URL=https://plugins.example.com
      - AUTH_TOKENS=your-secret-token
      - ADMIN_PASS=your-admin-password
      - SESSION_SECRET=your-session-secret
    restart: unless-stopped

volumes:
  registry-data:
```

### 4.3 使用外部 PostgreSQL

```yaml
services:
  registry:
    build: .
    ports:
      - "3000:3000"
    volumes:
      - registry-plugins:/app/data/plugins
    environment:
      - BASE_URL=https://plugins.example.com
      - AUTH_TOKENS=your-secret-token
      - DB_TYPE=postgresql
      - DB_URL=postgres://registry:secret@db:5432/plugin_registry
    depends_on:
      - db

  db:
    image: postgres:16-alpine
    environment:
      - POSTGRES_DB=plugin_registry
      - POSTGRES_USER=registry
      - POSTGRES_PASSWORD=secret
    volumes:
      - postgres-data:/var/lib/postgresql/data

volumes:
  registry-plugins:
  postgres-data:
```

## 5. 反向代理（HTTPS）

JetBrains IDE 要求 `url` 是 HTTPS。服务本身不处理 TLS。

### 5.1 Caddy（最简单，自动 HTTPS）

```
plugins.example.com {
    reverse_proxy localhost:3000
    request_body {
        max_size 100MB
    }
}
```

### 5.2 Nginx

```nginx
server {
    listen 443 ssl http2;
    server_name plugins.example.com;

    ssl_certificate /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    client_max_body_size 100m;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 300s;
    }
}
```

## 6. 访问日志

### 6.1 Hono Logger 中间件

```typescript
import { logger } from "hono/logger";

app.use("*", logger());
// 输出: <-- GET /updatePlugins.xml
//       --> GET /updatePlugins.xml 200 12ms
```

### 6.2 自定义日志格式

```typescript
app.use("*", async (c, next) => {
  const start = Date.now();
  await next();
  const ms = Date.now() - start;
  const { method } = c.req;
  const path = c.req.path;
  const status = c.res.status;

  // 跳过健康检查
  if (path === "/api/health") return;

  console.log(`${new Date().toISOString()} ${method} ${path} → ${status} (${ms}ms)`);
});
```

输出示例：
```
2026-02-21T10:00:00.000Z GET /updatePlugins.xml → 200 (5ms)
2026-02-21T10:00:01.000Z GET /plugins/com.example.myplugin/1.2.0 → 200 (12ms)
2026-02-21T10:01:00.000Z POST /api/plugins → 201 (850ms)
```

## 8. 备份

### 8.1 需要备份的数据

```
data/
├── registry.db      ← SQLite 数据库
└── plugins/         ← 插件 ZIP 文件
```

### 8.2 备份脚本

```bash
#!/bin/bash
BACKUP_DIR="/backup/plugin-registry/$(date +%Y%m%d)"
mkdir -p "$BACKUP_DIR"

# SQLite 在线备份（不影响服务运行）
sqlite3 /app/data/registry.db ".backup $BACKUP_DIR/registry.db"

# 插件文件
rsync -a /app/data/plugins/ "$BACKUP_DIR/plugins/"

echo "Backup completed: $BACKUP_DIR"
```

## 10. 数据库迁移

### 10.1 Drizzle Kit 迁移工具

项目使用 Drizzle Kit 管理数据库 schema 变更。

**安装（已包含在 devDependencies 中）：**

```bash
bun add -d drizzle-kit
```

### 10.2 生成迁移文件

当修改了 `src/db/schema.ts` 中的表结构后，运行：

```bash
bunx drizzle-kit generate
```

这会在 `drizzle/` 目录下生成 SQL 迁移文件（如 `0001_add_sessions_table.sql`），可以提交到版本控制。

### 10.3 开发模式：直接推送 schema

开发环境可以跳过迁移文件，直接同步 schema 到数据库：

```bash
bunx drizzle-kit push
```

> 注意：`push` 模式会直接修改数据库，不生成迁移文件，仅适用于开发环境。

### 10.4 生产模式：执行迁移

生产环境通过迁移文件逐步更新数据库：

```bash
bunx drizzle-kit migrate
```

### 10.5 首次启动自动建表

服务首次启动时，如果数据库文件不存在（SQLite）或表不存在，应用会自动创建所需的表结构。这通过 Drizzle ORM 的 `migrate()` 函数在启动时执行：

```typescript
import { migrate } from "drizzle-orm/bun-sqlite/migrator";

// 应用启动时自动执行迁移
migrate(db, { migrationsFolder: "./drizzle" });
```

## 11. 日志管理

### 11.1 日志输出策略

服务日志默认输出到 stdout，遵循 12-Factor App 原则：

```
2026-02-21T10:00:00.000Z GET /updatePlugins.xml → 200 (5ms)
2026-02-21T10:01:00.000Z POST /api/plugins → 201 (850ms)
2026-02-21T10:02:00.000Z DELETE /api/plugins/com.example.myplugin/1.0.0 → 200 (15ms)
```

### 11.2 Docker 环境日志收集

Docker 容器日志可通过标准方式查看：

```bash
# 查看实时日志
docker logs -f plugin-registry

# 查看最近 100 行
docker logs --tail 100 plugin-registry

# 按时间范围查看
docker logs --since 2026-02-21T00:00:00 plugin-registry
```

### 11.3 日志轮转

**Docker 日志驱动（推荐）：**

在 `docker-compose.yml` 中配置日志大小限制：

```yaml
services:
  registry:
    # ... 其他配置
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

**直接运行时使用 logrotate：**

```
/var/log/plugin-registry/*.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    copytruncate
}
```

### 11.4 关键事件审计日志

以下操作会输出结构化日志，便于审计追踪：

| 事件 | 日志示例 |
|------|----------|
| 插件上传 | `UPLOAD com.example.myplugin@1.0.0 by admin (1.2MB)` |
| 版本删除 | `DELETE com.example.myplugin@1.0.0 by admin` |
| 插件删除 | `DELETE_ALL com.example.myplugin (3 versions) by admin` |
| 登录成功 | `LOGIN admin from 192.168.1.100` |
| 登录失败 | `LOGIN_FAILED admin from 192.168.1.100` |

## 12. 故障恢复

### 12.1 SQLite 数据库损坏恢复

```bash
# 1. 检查数据库完整性
sqlite3 /app/data/registry.db "PRAGMA integrity_check;"

# 2. 如果损坏，尝试导出恢复
sqlite3 /app/data/registry.db ".dump" | sqlite3 /app/data/registry-recovered.db

# 3. 如果导出也失败，从备份恢复
cp /backup/plugin-registry/latest/registry.db /app/data/registry.db
```

### 12.2 数据目录丢失恢复

如果整个 `data/` 目录丢失：

1. **从备份恢复**（首选）：
   ```bash
   cp -r /backup/plugin-registry/latest/* /app/data/
   ```

2. **重建数据库**（插件文件仍在时）：
   ```bash
   # 重启服务，自动建表
   bun run src/index.ts
   # 重新上传所有插件（如有备份的 ZIP 文件）
   ```

3. **全新开始**：
   ```bash
   mkdir -p /app/data/plugins
   bun run src/index.ts  # 自动创建空数据库
   ```

### 12.3 服务无法启动排查清单

| 问题 | 检查方法 | 解决方案 |
|------|----------|----------|
| 端口被占用 | `lsof -i :3000` | 修改 `PORT` 或停止占用进程 |
| 数据目录权限 | `ls -la /app/data/` | `chmod 755 /app/data/` |
| 数据库被锁 | `fuser /app/data/registry.db` | 停止其他访问该数据库的进程 |
| 依赖缺失 | `bun install` | 重新安装依赖 |
| 环境变量未设置 | 检查 `ADMIN_PASS` 等必填项 | 补充缺失的环境变量 |
| SQLite 文件损坏 | `sqlite3 ... "PRAGMA integrity_check;"` | 见 12.1 SQLite 恢复 |

### 12.4 Docker 容器重启策略

docker-compose.yml 中已配置 `restart: unless-stopped`，容器异常退出后会自动重启。

手动操作：

```bash
# 查看容器状态
docker compose ps

# 重启服务
docker compose restart registry

# 查看容器退出原因
docker inspect --format='{{.State.ExitCode}} {{.State.Error}}' plugin-registry
```

## 13. 监控告警

### 13.1 健康检查端点

`GET /api/health` 返回服务状态：

```json
{
  "status": "ok",
  "uptime": "3d 12h 30m",
  "database": "connected",
  "version": "1.0.0"
}
```

可通过 HTTP 状态码判断服务是否正常（200 = 正常）。

### 13.2 Docker 健康检查

Dockerfile 中已配置：

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
    CMD wget -q --spider http://localhost:3000/api/health || exit 1
```

查看健康状态：

```bash
docker inspect --format='{{.State.Health.Status}}' plugin-registry
```

### 13.3 磁盘空间监控

插件文件会持续占用磁盘空间，建议监控：

```bash
# 查看数据目录大小
du -sh /app/data/

# 查看磁盘剩余空间
df -h /app/data/

# 通过 API 查看存储统计
curl -s http://localhost:3000/api/stats | jq '.totalStorageHuman'
```

建议当磁盘使用率超过 80% 时触发告警。

### 13.4 Uptime 监控方案

推荐使用轻量级的 uptime 监控工具：

**Uptime Kuma（推荐，自托管）：**

```yaml
# 添加到 docker-compose.yml
services:
  uptime-kuma:
    image: louislam/uptime-kuma:1
    ports:
      - "3001:3001"
    volumes:
      - uptime-data:/app/data
    restart: unless-stopped

volumes:
  uptime-data:
```

在 Uptime Kuma 中添加 HTTP 监控：
- URL: `http://registry:3000/api/health`
- 间隔: 60 秒
- 超时: 10 秒
- 预期状态码: 200

**简单脚本监控（无需额外工具）：**

```bash
#!/bin/bash
# cron: */5 * * * * /opt/scripts/check-registry.sh
HEALTH=$(curl -sf http://localhost:3000/api/health)
if [ $? -ne 0 ]; then
  echo "Plugin Registry is DOWN!" | mail -s "ALERT: Plugin Registry" admin@example.com
fi
```
